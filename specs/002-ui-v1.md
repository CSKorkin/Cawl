# Spec: Cawl UI V1

Companion to `specs/001-pairing-engine.md`. This spec covers the UI layer
that wraps the pure-TS engine. The engine is feature-complete through T13
(Easy + Medium AI); Hard AI (T14) is in flight and lands behind a feature
gate. This spec scopes the **full V1 UI surface** but is delivered in
phased, demoable slices — see _Phased delivery_ below.

## Objective

Cawl's purpose (per `CLAUDE.md`) is to teach competitive WTC pairing
decisions: pairing is a game of incomplete information played on top of a
slightly asymmetric shared belief matrix. The UI exists to surface that
mental model — not to be a slick game wrapper.

V1 ships two play modes:

- **Hot-seat**: two humans, one device. Control switches between teams via
  full-screen "pass the device" interstitials that enforce information
  hiding through UI flow, not just data hiding.
- **Single-player vs AI**: human plays one seat; the engine drives the
  other via `easyActor` / `mediumActor` / (later) `hardActor`. The AI
  commits before reveal — humans cannot peek.

Both modes support standard 0–20 and Atlas ordinal scoring, and either
seed-generated or user-entered matrices.

## Target users

Two players who want to practice WTC pairing without scheduling a real
team match — solo against an algorithm, or together on one laptop.
Familiar with the WTC rules; the UI assumes the domain glossary
(defender, attackers, refusal, table choice token, scrum) without
explanation.

## Tech stack

Per `CLAUDE.md` defaults; no deviations:

- **Framework**: React + TypeScript via Vite
- **Styling**: Tailwind. Color bands map to utility classes; no custom CSS
  for the matrix coloring.
- **State**: Zustand. The engine FSM is already a state machine — Zustand
  wraps it as a single store with `dispatch(action)` calling
  `applyAction(state, action)` and persisting the result.
- **Persistence**: `localStorage`. Serialize on every `applyAction`
  return; rehydrate on mount. State is JSON round-trippable by engine
  invariant — already enforced by the engine test suite.
- **Routing**: not needed for V1. Three views (`Setup`, `Play`,
  `GameOver`) selected by store state, no URL routing.
- **Tests**: Vitest + React Testing Library. Engine tests already pass on
  Vitest 3.

**Zero new engine dependencies.** Everything UI-side imports from
`src/engine/index.ts` (curated public surface, finalized in T15) or, until
that lands, the explicit module paths used by the demos.

## Application views

Three top-level views, selected purely by store state:

| Store state              | View       | Purpose                                          |
|--------------------------|------------|--------------------------------------------------|
| no engine state, no game | `Setup`    | Configure mode, scoring, matrix source, rosters  |
| engine state, mid-game   | `Play`     | Main play screen                                 |
| `phase === GAME_COMPLETE`| `GameOver` | Final slate, log replay, "play again" button     |

Hot-seat layers a fourth modal-only view, `Interstitial`, on top of `Play`
during team-handoff moments.

### Setup view

Single screen, sectioned:

1. **Mode**: radio — "Hot-seat" / "Single-player vs AI". For SP, a tier
   sub-radio appears: "Easy" / "Medium" / "Hard (coming soon)". Hard is
   visually present but disabled until T14 lands.
2. **Scoring**: radio — "Standard (0–20)" / "Atlas (1–5 ordinal)".
3. **Matrix source**: radio — "Generated" / "Entered". For Generated, the
   system auto-generates a seed and presents a "Re-roll" button. 
   For Entered, an 8×8 grid of inputs (typed for the chosen scoring mode).
   Both options write to the same engine `Matrix` shape; the toggle is a
   pre-engine concern.
4. **Rosters**: two side-by-side panels, one per team. Each is a column
   of 8 slots. Each slot is a faction picker (logo + name) sourced from
   `public/logos/` — see _Faction catalog_ below. If using "Generated", these
   are picked at random with no duplicates. If using "Entered", they are
   entered at the same time as the matrix from a drop-down. 
5. **Start**: button enabled only when all 16 slots are picked AND (if
   "Entered") the matrix is fully populated.

### Play view

Persistent layout while phase is anything mid-game (`ROUND_*` /
`SCRUM.*` except `GAME_COMPLETE`):

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Header: Mode tag │ Phase chip │ Token chip (◯ A / ◯ B) │ Round badge   │
├──────────┬───────────────────────────────────────────────┬──────────────┤
│          │                                               │              │
│ Team A   │            8×8 Matrix       │ ROW            │ Team B       │
│ roster   │       (current viewer's view) │ avg          │ roster       │
│          │      column = opposing army │ (own-army    │              │
│ 8 slots: │      row = own team's army  │  expected   │ 8 slots:     │
│ logo +   │      cell = expected score, │  matchup     │ logo + name  │
│ name +   │             color-banded    │  vs remaining│ + status     │
│ status   │                             │  opp pool)   │              │
│          ├──────────────────────────────┴──────────────┤              │
│          │ COL avg (our team's expected score vs each   │              │
│          │  remaining opp army across our remaining     │              │
│          │  pool)                                       │              │
├──────────┴───────────────────────────────────────────────┴──────────────┤
│ Step prompt + action zone:                                              │
│   "Team A — pick your defender."  [confirm only when slot selected]     │
├─────────────────────────────────────────────────────────────────────────┤
│ Log panel (collapsible, append-only)                                    │
└─────────────────────────────────────────────────────────────────────────┘
```

**Whose view does the matrix render?** The team currently moving — `viewFor(state, currentMover)`. In hot-seat that switches between picks; in SP it's always the human's seat. The opposing team's roster on screen is shown without any matrix data leaking — armies in opp's roster are listed but their matchup scores are never displayed.

**Slot status badges** (small chips on each roster slot):
- `pool` (default — eligible for picks)
- `def-locked` (current defender)
- `attackers-locked` (one of two attackers)
- `paired` (locked into a pairing — shows table number once assigned)
- `refused` (opp refused this attacker; back in pool)

**Row and column averages**: the matrix has two margin strips that
display per-army averages computed live from the current viewer's matrix:

- **Right margin (row averages)**: one cell per row. Value = mean of
  `myView[i][j]` for j ∈ remaining `oppPool`. This is "what the viewer
  expects their army `i` to score on average against any eligible
  opposing army." Useful for defender decisions (high row-avg = strong
  defender candidate).
- **Bottom margin (column averages)**: one cell per column. Value = mean
  of `myView[i][j]` for i ∈ remaining own `pool` (excluding the viewer's
  current defender once locked, to mirror attacker eligibility).
  This is "what the viewer expects their team to score on average against
  opposing army `j`." Useful for sizing up opp defenders.

Both averages **recompute as pools shrink** — the static matrix cells
keep their values, but the margin averages reflect remaining
matchups, so they stay decision-relevant in late rounds and the scrum.

Averages are shown as decimals with one fractional digit (e.g. `11.4`),
color-banded by the same scheme as cells (rounded to nearest integer for
band lookup in standard mode; nearest tier for atlas). They are
viewer-specific aggregates of the viewer's own matrix only — no opp-view
data leaks. Bottom margin uses the same column-rendering as the matrix;
right margin sits visually adjacent to the corresponding row.

The averages are derived purely from `viewFor(state, currentMover)` —
the existing information-hiding invariant covers them automatically.

**Selection mechanics**:
- Click an own-team slot in `pool` → selects.
- Click again or click another → re-selects.
- "Confirm" button (or Enter key) → dispatches the action.
- Matrix cells are display-only in the Play view (no click-to-select).
  Selection happens on the roster, not on the matrix.

**Reveal animation**: when both pendings collapse into `revealed`, the
two newly-revealed slots flash for ~600ms. Out of scope for early phases;
land in polish phase.

### GameOver view

- Final 8-game slate, table-ordered (matches the m4/m5 demos' format).
- Each row: T# · round badge · matchup logos+names · defender · A-score
  (B-score) · — with totals at the bottom and a predicted-result verdict
  ("Team A wins by N").
- Log replay (read-only collapsible panel).
- "Play again" button: returns to `Setup` (preserving last-used config as
  a convenience. For generated, this retains the same seed by default).
- "Export transcript" button: downloads the log + final state as a JSON
  file (engine state is JSON-round-trippable; this is a one-liner).

### Interstitial (hot-seat only)

Full-screen modal. Triggered automatically by the store between every
team-handoff moment in hot-seat:

- Between defender lock-ins (A picks → interstitial → B picks → reveal)
- Between attacker lock-ins
- Between refusal lock-ins
- (Table picks are not hidden — they're public information per the spec
  log.)

Content: large `[Pass to Team B]` button; nothing else on screen — no
matrix, no log. Tap → control switches; matrix/roster re-render with
new viewer's data.

In SP mode, the interstitial **is not shown** — the AI's lock-in fires
synchronously inside `applyAction` and the human's view never had a
chance to peek (the engine's information-hiding invariant is the only
guarantee needed).

## Information hiding

The engine already enforces information hiding by construction
(`viewFor(state, seat)` strips opposing pendings; AI Actors take only
`TeamView`). The UI extends this to the *flow* layer:

1. **Hot-seat**: interstitials gate every team handoff during secret
   choices. The store's `dispatch` wrapper holds reveals until the
   interstitial is dismissed.
2. **SP vs AI**: the AI's pick is dispatched synchronously after the
   human's, inside the same store action. The human never sees an
   intermediate state where their pending is exposed but the AI's isn't —
   it goes straight from "human pending only" to "both revealed."
3. **No "view as opponent"**: even in hot-seat replay/spectator mode,
   the UI never exposes opp's matrix. (The engine state has both views;
   the UI must only render `viewFor(state, currentMover)`.)

## Architecture

```
src/
├── engine/                  # untouched by this spec
├── factions.ts              # catalog (id, displayName, logoPath)
├── store/
│   ├── gameStore.ts         # Zustand store wrapping engine
│   ├── persistence.ts       # localStorage rehydrate/serialize
│   └── selectors.ts         # derived state for UI components
├── components/
│   ├── Setup/
│   │   ├── ModePicker.tsx
│   │   ├── ScoringPicker.tsx
│   │   ├── MatrixSourcePicker.tsx
│   │   ├── MatrixEntryGrid.tsx       # for "Entered" mode
│   │   ├── RosterPicker.tsx
│   │   └── FactionPicker.tsx         # single-slot picker
│   ├── Play/
│   │   ├── PlayScreen.tsx
│   │   ├── Matrix.tsx                # 8×8 viewer
│   │   ├── Roster.tsx                # team's 8 slots
│   │   ├── ArmySlot.tsx              # one logo+name+status chip
│   │   ├── TokenChip.tsx
│   │   ├── PhaseChip.tsx
│   │   ├── StepPrompt.tsx            # current-step CTA
│   │   └── LogPanel.tsx
│   ├── GameOver/
│   │   ├── GameOverScreen.tsx
│   │   └── FinalSlate.tsx
│   ├── Interstitial.tsx
│   └── App.tsx                       # view router
├── hooks/
│   └── useAIDriver.ts                # SP mode: dispatches AI moves on phase change
├── main.tsx
└── index.css                         # Tailwind directives
```

**Component boundaries**:
- Components read store state via Zustand selectors. No prop-drilling
  for store data.
- `Matrix` consumes `viewFor(state, currentMover)` — never reaches into
  raw `state.matrix.viewA/viewB`.
- `useAIDriver` watches phase + active mover; when it's the AI's seat,
  computes the action via the chosen Actor and dispatches synchronously.

## Phased delivery

The full V1 surface above ships in 7 phases. Each phase is independently
demoable and reviewable. Each builds on the engine + previous phase
(no rework). Phases are sized for one focused session each.

### Phase U1 — Scaffold + Setup screen (static)

**Deliverables**:
- Vite + React + TS + Tailwind project scaffolded under existing
  `src/` (engine stays put).
- `factions.ts` catalog. Logos restored to `public/logos/` from git
  history.
- `Setup` view: mode + scoring + matrix-source toggles + faction
  pickers. No play yet — "Start" button is wired but goes nowhere.
- Tests: catalog completeness; `Setup` renders 23 factions;
  duplicate-suffixing logic.

**Acceptance**: `npm run dev` opens the Setup screen; selecting a mode +
scoring + 16 factions + a seed enables the Start button. Engine tests
still green.

### Phase U2 — Play screen (single-player vs Easy, generated matrix)

**Deliverables**:
- Zustand store wrapping engine; `dispatch(action)` calls
  `applyAction`, persists result.
- `Play` view skeleton: matrix renderer, both rosters, step prompt,
  log panel (text-only, no styling polish). Token chip + phase chip
  in header.
- `useAIDriver` for SP mode: dispatches AI's lock-in synchronously.
- Easy AI plays through to `GAME_COMPLETE` (no end-game screen yet —
  shows a placeholder).
- Tests: store wraps engine; matrix renders correct view per
  `viewFor`; one full-game smoke test through React Testing Library.

**Acceptance**: human can complete a full game vs Easy on a generated
matrix, with all 8 pairings logged.

### Phase U3 — GameOver screen

**Deliverables**:
- `GameOver` view with final slate (table-ordered, scores, predicted
  verdict). Mirrors m5 demo's output format.
- "Play again" returns to Setup.
- "Export transcript" downloads JSON.
- Tests: GameOver renders 8 rows; export round-trips.

**Acceptance**: a single-player vs Easy game completes end-to-end with
a presentable final-slate screen.

### Phase U4 — Hot-seat with information-hiding interstitials

**Deliverables**:
- Hot-seat mode wired in store: dispatch holds reveals until both
  teams have committed AND the interstitial is dismissed.
- `Interstitial` component: full-screen, "Pass to Team X" CTA.
- Matrix re-renders on team switch.
- Tests: invariant test that during hot-seat play, `viewFor(B)` is
  never rendered while `pendingA` is set; interstitial appears between
  every secret-choice handoff.

**Acceptance**: two humans can play through a full hot-seat game with
no information leaking across the device.

### Phase U5 — User-entered matrix

**Deliverables**:
- "Entered" matrix-source flow: 8×8 grid of inputs, mode-aware
  (integer 0–20 for standard; tier dropdown for atlas).
- Validation: range / tier checks; required-fill before Start
  enables.
- Optional convenience: "fill from clipboard / CSV paste" for users
  who track their matrix in a spreadsheet.
- Tests: entry validates per mode; engine accepts entered matrix as
  `viewA` (engine derives `viewB` via inversion + variance from the
  same RNG path).

**Acceptance**: a user can paste their team's training matrix and play
a full game on it.

### Phase U6 — Polish

**Deliverables**:
- Reveal animations (~600ms flash on collapsed slots).
- Token-handoff animation between rounds.
- Log panel styling (collapsible, color-banded entries, hover
  reveals).
- Persistence: full reload preserves in-flight game state.
- Empty-state styling for Setup; faction-picker keyboard nav.
- Atlas mode visual differentiation (different cell formatting for
  `2.5` / `3.5`).
- Demo screenshots / a brief README walkthrough.

**Acceptance**: ships looking like a real product, not a wireframe.

### Phase U7 — Hard AI integration (after T14)

**Deliverables**:
- Unlock the "Hard" radio in Setup once T14 lands.
- Per-decision performance budget (10ms cap from spec); if exceeded,
  surface a brief "AI thinking…" indicator.
- Tests: hard-vs-easy and hard-vs-medium runs through the UI complete
  legally.

**Acceptance**: the spec's success criterion 6 ("Medium beats Easy >70%
and Hard exists") is reachable end-to-end through the UI.

## Code style

Match the engine's house style:

- Strict TypeScript. `strict: true`, `noUncheckedIndexedAccess: true`,
  `exactOptionalPropertyTypes: true`. No `any` without an inline reason.
- Function components only. No class components.
- One Zustand store. No Redux, no Context for state.
- Tailwind utility classes. Component-local CSS modules only when
  utilities can't express it (rare).
- Component file = component name. Co-locate tests as `*.test.tsx`.
- No god components. Each component fits on one screen.
- Naming reflects domain (`defender`, `attackers`, `refused`,
  `tableChoiceToken`) — same vocabulary the engine uses.

Example (not actual file — just the shape):

```tsx
import { useGameStore } from '../store/gameStore';
import { viewFor } from '../engine/state';

export function StepPrompt() {
  const { state, currentMover, dispatch } = useGameStore();
  const view = viewFor(state, currentMover);
  // ... render the action button per state.phase
}
```

## Testing strategy

| Layer | Framework | What's tested |
|---|---|---|
| Store | Vitest | dispatch wrapping, persistence rehydrate, info-hiding gating |
| Components | RTL | rendering per state, click handlers, keyboard a11y |
| Integration | Vitest | full-game smoke (one human + one AI) through the React tree |
| Engine | (already exists) | unchanged |

Coverage gates: store ≥90% line/branch; components no gate (interesting
cases caught by integration tests). Engine gates as before.

**The information-hiding test**: a property-style test that drives a
random hot-seat game and asserts at every render that the rendered
matrix corresponds to `viewFor(state, currentMover)` and never leaks
opp's pending. This is the UI counterpart of the engine's 200-seed
information-hiding property test.

## Boundaries

**Always do**:
- Read engine state via `viewFor(state, currentMover)` from any
  rendering component. Never reach into `state.matrix.viewA/viewB`
  directly.
- Persist on every dispatch.
- Run the engine test suite before merging — UI changes shouldn't
  affect engine, but a regression-canary diff makes accidents loud.

**Ask first**:
- New runtime dependencies (anything beyond React / Vite / Tailwind /
  Zustand).
- Schema changes to `Faction` or the catalog.
- Changes to engine `index.ts` exports (UI shouldn't drive engine API
  changes; if it needs something, surface and discuss).

**Never**:
- Render `state.matrix.viewB` while `state.tokenHolder === 'A'` or
  current mover is A. The "view-by-seat" invariant is the whole point
  of this UI.
- Skip the interstitial in hot-seat to "speed things up." If we're
  going to skip it ever, it's a deliberate spec change, not a
  shortcut.
- Use `Math.random` / `Date.now` in components or store. RNG is
  engine-owned; UI is deterministic.
- Build network multiplayer. Out of scope; V2.

## Success criteria

- A user can complete a full single-player-vs-Easy game on a
  generated matrix, end-to-end through the UI, in under 5 minutes of
  interaction.
- A user can complete a full hot-seat game, with at least one
  observable info-hiding interstitial per round, with no leaked opp
  matchup data.
- A user can enter a custom matrix and play through it.
- After page reload at any mid-game phase, the game resumes exactly
  where it was — same phase, same pendings (per current viewer),
  same log, same RNG state.
- All engine tests still pass; UI tests pass; typecheck clean;
  `npm run build` succeeds.
